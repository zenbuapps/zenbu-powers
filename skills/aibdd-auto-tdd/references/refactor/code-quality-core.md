# 程式碼品質核心規則（語言無關）

> 本檔提供 **Refactor 階段共用的品質規則**，所有語言 reference 共享。
> 語言特化的程式碼範例與檢查清單見 `references/code-quality/{csharp|php|typescript}.md`。

## 重構原則

1. **測試保護下重構** — 每次修改後重跑測試，確保行為不變
2. **小步驟** — 每次只改一件事，確認通過後再繼續
3. **不改外部行為** — 重構不加新功能、不改 API 契約
4. **三次以上再抽** — 重複兩次可接受，三次以上才抽取

## 品質檢查清單

### 1. SOLID 原則

| 原則 | 檢查項目 |
|------|---------|
| S（單一職責） | 每個 class/function 只做一件事？ |
| O（開放封閉） | 新增行為不需修改既有程式碼？ |
| L（里氏替換） | 子類別可安全替換父類別？ |
| I（介面隔離） | 介面不強迫實作不需要的方法？ |
| D（依賴反轉） | 高層模組不依賴低層模組？ |

### 2. DRY（Don't Repeat Yourself）

- 相同邏輯出現 3+ 次？→ 抽取共用函數
- 相同常數散落多處？→ 集中定義
- 注意：**不過度抽象**，3 行重複比一個看不懂的抽象好

### 3. 架構分層

```
API (Controller/Router) → Service → Repository → Model
```

- Controller 只做 HTTP 轉換（Request → 呼叫 Service → Response）
- Service 實作業務邏輯
- Repository 負責資料存取
- Model 定義資料結構
- **各層職責分明，不跨層**

### 4. 命名清晰

- 函數名表達意圖（`update_video_progress` 而非 `process`）
- 不用縮寫（`repository` 而非 `repo`，除非是專案慣例）
- 布林變數用 is/has/can 開頭

### 5. Step Definition 整理

- 檔案組織：按 subdomain + handler 類型分目錄
- Import 排序：標準庫 → 第三方 → 本地
- 共用 helper 提取：狀態映射（中文→enum）等

### 6. Meta 清理

- 移除所有 TODO 註解（紅燈階段的遺留）
- 保留業務邏輯註解
- 移除無用 import
- 移除已被註解的程式碼

## 小步驟循環

```
1. 確認測試全數通過（起始綠燈）
2. 識別一個改善點（依上方優先級）
3. 執行重構
4. 重新測試
   ├─ 通過 → 提交，回到 2
   └─ 失敗 → 還原，換另一個改善點
5. 無更多改善點 → 結束
```

## 不做的事

- 不加新功能
- 不改測試（除非重構測試程式碼本身）
- 不改 API 契約或 DB schema
- 不做效能優化（除非明顯問題）
